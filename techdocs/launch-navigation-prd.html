Version: 1.2 Date: 2026-07-21 Status: Implemented (JL-277, JL-282, JL-283) Source: origin/main @ 572d340 Owner: sirisha@sedintechnologies.com

1. Summary

The application's left-hand navigation had grown to roughly 28 items across three groups (primary, product, utility) plus dynamically contributed plugin links. For the launch we present a curated surface: the Projects entry, the admin/oversight product tools (Teams, Users, Activity, Workflows, Audit Log), and the full Filters box of reporting & views (Filters, Dashboards, Portfolio, Report Builder, Advanced Roadmap, Shared Dashboards, Cross-Project Boards). The remaining product items and the plugin nav are hidden from the sidebar while remaining fully functional at their routes (nav-only curation), so the reduction is presentation, not feature removal, and is fully reversible via a single flag.

2. Goals & Non-Goals

Goals

  • Present a focused, uncluttered navigation for launch users centered on administration & oversight (Users, Teams, Audit Log) and core work surfaces (Filters, Activity, Workflows).
  • Make the curation reversible — a single flag restores the full navigation with no code archaeology.
  • Preserve all existing role/permission gating and cause no regressions to hidden features.

Non-Goals

  • Removing routes or features. Hidden pages (Dashboards, Automation, Releases, etc.) remain reachable by URL and continue to work.
  • Reworking the destination pages themselves (each already exists and ships on main).
  • Server-side authorization changes — this is a client-side navigation presentation change.

4. What is hidden (and why it's safe)

The following remain routable by URL but are removed from the sidebar for launch:

  • Primary: Recent. (Projects is kept — see §3.)
  • Product: Webhooks, Inbound Email, Automation, Marketplace, Releases, Goals, Assets, Knowledge Base, Help Center, Queues, Incidents, Apps, BI Export.
  • Plugin nav: the entire dynamically-contributed nav section and its divider.

The Utility group (the Filters box) is now shown in full — see §3.

Resolved (JL-282): the initial curation (JL-277) also hid the Projects block, removing the in-sidebar entry point to boards/backlogs. Projects was subsequently added back to LAUNCH_NAV, so the launch sidebar retains it while Recent stays hidden.

5. Implementation

Single file: src/components/layout/Sidebar.jsx. A reversible allow-list drives everything:

  • LAUNCH_SIDEBAR = true — master switch; set false to restore the full navigation.
  • LAUNCH_NAV = ['Projects','Filters','Teams','Users','Activity','Workflows','Audit Log','Dashboards','Portfolio','Report Builder','Advanced Roadmap','Shared Dashboards','Cross-Project Boards'] — the curated set (Projects added in JL-282; the six other utility items restored in JL-283).
  • launchFilter is applied to the primaryItems, productItems, and utilityItems arrays after their existing permission spreads, so role gating is preserved.
  • Empty-state chrome is guarded on primaryItems.length > 0: the first <nav>, its divider, and the "Show Projects" restore button don't render when the primary group is empty. With Projects kept, the primary nav landmark now renders.
  • The plugin nav section renders only when !LAUNCH_SIDEBAR.

Tests: src/test/Sidebar.launch.test.jsx — asserts the six product/utility labels plus Projects render with correct hrefs, that the remaining non-launch labels are absent, and that the primary nav landmark renders. 4/4 passing.

Delivery: PR #36 (curation, JL-277 · app JL-51), PR #37 (keep Projects, JL-282 · app JL-52), and PR #38 (restore Filters-box utility items, JL-283 · app JL-53), squash-merged to main (commit 572d340).
